# Project Overview

## **What is a Project?**

A **Project** in Skala is a way to organize **similar tasks** to ensure consistency in **parameters, instructions, and workflow settings**. Projects allow you to:

- Define **common parameters** that tasks will inherit.
- Control **which workers** can work on specific task types.
- Manage **project-wide versioning** for easier tracking.

To associate a task with a project, include a **project parameter** when creating a task:

```json filename="JSON"
{
  "project": "my_project"
}
```

> **Note:**

- Projects created using your **test API key** can only be used for **test tasks**.
- Projects created using the **live API key** can only be used for **live tasks**.

---

## **Using Project-Level Parameters**

Skala **uses Projects** to organize **Tasks**, the fundamental unit of work. Each **task type** (e.g., **Image Annotation, Text Collection, LiDAR Annotation**) has **customizable parameters** that guide task creation and response formatting.

These parameters can be stored at:

1. **Task Level**
2. **Project Level**
3. **Hybrid Approach** (combination of both)

---

## **Task vs. Project-Level Parameters**

### **Task-Level Parameters (Task Only)**

- Parameters are **fully defined** when creating each task.
- The project is **only used for grouping tasks**.
- Task-level parameters **override** project settings.

### **Project-Level Parameters (Project Only)**

- All **task definitions** are stored at the **project level**.
- Task creation requires **only essential parameters** (e.g., `attachment`).
- Tasks **inherit** parameters automatically.

### **Hybrid Approach**

- Some parameters are **defined at the project level**.
- Others are **customized per task**.
- Allows **project versioning** while keeping **task flexibility**.

---

## **Project-Level Features**

### **1️⃣ Versioning**

- Skala maintains a **history** of all past project versions.
- Tasks reference **specific versions** to ensure consistency.
- Each task includes `project_param_version`, indicating the **parameter version used**.

### **2️⃣ Separation of Concerns**

- **Task submission remains simple**, without needing full project details.
- The **project dictates labeling rules** automatically.
- **Updates to the project** do not require modifying individual tasks.

### **3️⃣ Default Values**

- Project-level parameters serve as **default values** for tasks.
- Helps ensure **consistent instructions** and **default settings** across all tasks.

---

## **How to Use Project-Level Parameters**

### **At the Project Level**

✅ Use the **Update Parameters API** to define project settings.  
✅ Retrieve **paramHistory** to track **parameter changes**.

### **At the Task Level**

✅ Tasks **inherit** project-level parameters automatically.  
✅ Task-level parameters **override** project parameters **except** for `instruction`, which combines both.  
✅ Use `project_param_version` to reference **past versions**.

---

# **Using Project Ontologies**

## **What is an Ontology?**

Skala **uses Ontologies** to:

- Maintain **complex taxonomies**.
- Provide **detailed labeling guidance**.
- Track **ontology changes over time**.

Ontologies can serve as **a replacement or add-on** to project instructions.

---

## **Ontology Features**

### **1️⃣ Versioning**

- Skala maintains a **history of ontology versions**.
- Labelers can **compare past versions** to track changes.

### **2️⃣ Descriptions**

- Each **label choice** can include **detailed descriptions**.
- Helps labelers understand **how to categorize** objects correctly.

---

## **How to Use Ontologies**

### **At the Project Level**

✅ Use the **Update Ontology API** to manage project ontologies.  
✅ Retrieve **ontologyHistory** to track **ontology changes**.

### **At the Task Level**

✅ Tasks automatically **use the latest ontology version**.  
✅ Ontologies **do not inherit project parameters** by default.

## **Example Ontology Definition**

```json filename="ontology.json"
{
  "ontology": [
    "Road",
    {
      "choice": "Vehicle",
      "description": "A means of carrying or transporting material.",
      "subchoices": ["Car", "Truck", "Train", "Motorcycle"]
    },
    {
      "choice": "Pedestrian",
      "subchoices": [
        "Animal",
        { "choice": "Ped_HeightOverMeter", "display": "Adult" },
        { "choice": "Ped_HeightUnderMeter", "display": "Child" }
      ]
    }
  ]
}
```

## **Summary**

| **Feature**                  | **Details**                                  |
| ---------------------------- | -------------------------------------------- |
| **Project Usage**            | Organizes tasks, applies shared parameters.  |
| **Task-Level Parameters**    | Defined when creating tasks.                 |
| **Project-Level Parameters** | Inherited from project, versioned.           |
| **Ontology Usage**           | Defines labels and rules for tasks.          |
| **Versioning**               | Available for **Projects** & **Ontologies**. |
